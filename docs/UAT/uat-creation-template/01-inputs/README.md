# 01-inputs ディレクトリについて

## 概要

このディレクトリには、UAT仕様書作成に必要な設計書をMarkdown形式で配置します。

## ファイルの配置方法

### 方法1: 手動で配置（推奨：設計書が整理済みの場合）

必要な設計書を直接このディレクトリに配置してください。

### 方法2: 自動選別コマンドを使用（推奨：docs/ ディレクトリから選別する場合）

プロジェクトの `docs/` ディレクトリから必要なファイルのみを自動的にコピーできます。

#### 手順

1. **Claude Codeで選別コマンドを実行**

```bash
claude
```

Claude Codeのプロンプトで:

```
/input-prepare
```

2. **自動的に分析・選別・コピー**

以下が自動的に実行されます:
- プロジェクトの `docs/` ディレクトリ（テンプレートからの相対パス `../../`）内の全Markdownファイルを分析
- `docs/UAT/` ディレクトリは分析対象から除外（循環参照防止）
- file-analyzerサブエージェントによるUAT観点での有用性評価
- 必要なファイルのみを `01-inputs/` にコピー

#### 選別基準

- **High Priority**: ユーザー操作、画面仕様、業務フロー、API設計
- **Medium Priority**: システム概要、データ仕様、セキュリティ
- **Low Priority**: 非機能要件、技術仕様
- **除外**: 開発環境設定、プロジェクト管理、テスト仕様

#### 利点

- **処理時間短縮**: 不要な設計書を読み込まない
- **トークン削減**: 関連する設計書のみを処理
- **精度向上**: ノイズとなる情報を削減

## 必須ファイル

### 機能一覧表.md（推奨）

全機能の一覧を記載したファイルです。以下の情報を含むことを推奨します:

- **機能ID**: 一意の機能識別子（例: CHAT-01, ATTACH-01）
- **機能名**: 機能の名称
- **実装状況**: 機能の実装状態（例: 実装済み）

**フォーマット例:**

```markdown
# 機能一覧表

| 機能ID | カテゴリ | 機能名 | 実装状況 |
|--------|----------|--------|---------|
| CHAT-01 | チャット | テキストメッセージ送受信 | 実装済み |
| ATTACH-01 | 画像添付 | 画像アップロード | 実装済み |
| AUTH-01 | 認証 | ユーザー認証 | 実装済み |
```

## その他の設計書

以下のような設計書を配置してください:

### 1. システムアーキテクチャ

- ファイル名例: `architecture.md`
- 全体構成、技術スタック、コンポーネント構成を記載

### 2. API仕様書

- ファイル名例: `api.md`
- エンドポイント、リクエスト/レスポンス形式、エラーハンドリングを記載

### 3. データ設計書

- ファイル名例: `database.md`
- テーブル構造、フィールド定義、制約条件を記載

### 4. セキュリティ設計書

- ファイル名例: `security.md`
- 認証・認可、RLS、レート制限を記載

### 5. 機能別設計書

- ファイル名例: `attachments.md`
- 個別機能の詳細設計を記載

## ファイル命名規則

### 推奨される命名規則

1. **機能名を含める**: `attachments.md`
   - 機能との関連付けが明確になります

2. **カテゴリを含める**: `admin/grant.md`
   - 設計書の種類が明確になります

3. **わかりやすい名前**: `e2e-manual-test-plan.md`
   - 内容が推測できる名前にします

### 避けるべき命名

- 空白スペースのみの名前
- 特殊文字を多用した名前
- 日本語以外の言語との混在（可能な限り統一）

## Markdown形式のガイドライン

### 基本構造

```markdown
# タイトル

## セクション1

内容...

### サブセクション1.1

詳細...

## セクション2

内容...
```

### 表の記載

```markdown
| 項目名 | 説明 | 備考 |
|--------|------|------|
| 項目1  | 説明1 | 備考1 |
| 項目2  | 説明2 | 備考2 |
```

### リストの記載

```markdown
- 項目1
- 項目2
  - サブ項目2.1
  - サブ項目2.2
- 項目3
```

### コードブロック

```markdown
\`\`\`json
{
  "key": "value"
}
\`\`\`
```

## 配置例

### 方法1: 手動配置の場合

```
01-inputs/
├── 機能一覧表.md              # 機能一覧
├── api.md                     # API仕様
├── architecture.md            # システムアーキテクチャ
├── database.md                # データ設計
├── security.md                # セキュリティ設計
├── attachments.md             # 画像添付機能設計
└── acceptance.md              # 受入基準
```

### 方法2: `/input-prepare` 使用の場合

**実行前:**

```
marubo_ai/
├── docs/                       # プロジェクトのドキュメント（ソース）
│   ├── api.md
│   ├── architecture.md
│   ├── database.md
│   ├── security.md
│   ├── admin/
│   ├── ai-note/
│   ├── UAT/                    # ← 除外対象
│   │   └── uat-creation-template/
│   │       └── 01-inputs/      # ← ターゲット（空）
│   └── ... (その他のドキュメント)
```

**`/input-prepare` 実行後:**

```
docs/UAT/uat-creation-template/
└── 01-inputs/                  # 選別されたファイルのみ
    ├── README.md               # 自動生成（分析結果付き）
    ├── api.md                  # High Priority
    ├── architecture.md         # Medium Priority
    ├── database.md             # Medium Priority
    └── ... (必要なファイルのみ)
```

**メリット:**
- `docs/` 内の全ファイルからUATに必要なもののみを選別
- `docs/UAT/` は自動的に除外
- 元の `docs/` ファイルは変更なし（コピーのみ）

## 注意事項

1. **Markdown形式必須**: すべてのファイルは `.md` 拡張子にしてください
2. **文字コード**: UTF-8で保存してください
3. **画像ファイル**: 可能な限りテキストで記載し、画像は最小限にしてください
4. **ファイルサイズ**: 1ファイルあたり10MB以下を推奨します

## よくある質問

### Q: Excel形式の設計書しかない場合は？

A: Markdown形式に変換する必要があります。以下のツールが利用可能です:
- `tools/python/excel_to_markdown.py` を使用
- オンラインツールで変換
- 手動でコピー&ペースト

### Q: PDFファイルは使えますか？

A: いいえ、Markdown形式(.md)に変換する必要があります。

### Q: 設計書がない業務はどうすればいい？

A: 業務一覧表には記載し、設計書ファイルは不要です。
   スコープ定義書には業務名のみが記載されます。

### Q: 複数のサブディレクトリを作成できますか？

A: はい、可能です。例:
```
01-inputs/
├── 機能一覧表.md
├── api/
│   ├── api.md
│   └── attachments.md
└── design/
    ├── architecture.md
    └── database.md
```
