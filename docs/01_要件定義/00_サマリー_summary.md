## 1. プロジェクト概要

### 1.1 プロダクト基本情報

**ARDORS（アーダース）** は、AIを搭載した個人向け生産性・タスク・プロジェクト・習慣・内省の統合管理パートナーアプリである。

| 項目 | 内容 |
|------|------|
| プロダクト名 | ARDORS（アーダース） |
| カテゴリ | 個人向け生産性管理SaaS |
| 提供形態 | Webファースト（レスポンシブ対応） |
| 開発形態 | ソロ開発・個人プロジェクト（SaaS展開を見据える） |
| ホスティング | Vercel Hobby（無料枠） |
| DB / Auth | Supabase（PostgreSQL + Auth + RLS） |

### 1.2 コアバリュー

> **「ARDORSだけ開けば1日が回る」**

タスク、スケジュール、目標、習慣、内省のすべてを1つのアプリに統合する。ユーザーは複数のツールを行き来する必要がなくなる。

### 1.3 AIとユーザーの関係

ARDORSにおけるAIの役割は「提案するAI、承認するユーザー」である。AIがタスクの整理・スケジュール生成・振り返り分析・行動提案を行い、ユーザーは1タップまたは短い対話で承認・修正する。AIが強制・決定することはなく、最終判断は常にユーザーが行う。

また、建設的AIの原則に基づき、AIは「褒めるだけのアシスタント」ではなく、データに基づいた率直なフィードバックを提供する。必要であればユーザーの行動を修正するような指摘も行う。

### 1.4 技術スタック概要

| カテゴリ | 技術 |
|----------|------|
| フレームワーク | Next.js 15 App Router + TypeScript |

| 状態管理 | Zustand（クライアント）+ TanStack Query（サーバー） |
| フォーム | React Hook Form + Zod |

| テスト | Vitest + Testing Library + Playwright |

## 2. 解決する課題（8つのペインポイント）

ARDORSは、マルチプロジェクトを抱える個人が日常的に直面する以下の8つのペインポイントを解決することを目的とする。

### ペインポイント一覧

| # | ペインポイント | 概要 | ARDORSの解決策 |
|---|--------------|------|---------------|
| P-1 | **情報の分散** | タスク・スケジュール・目標・習慣がバラバラなツールに散在している。全体像が把握できず、「どこに何があるか」を管理するだけで疲弊する | 1つのアプリにすべてを統合。ダッシュボードで全情報を一覧できる |
| P-2 | **一覧性の欠如による迷い** | 多数のプロジェクトと締切を同時に抱えると認知負荷が爆発する。「今日何をすべきか」を判断するだけで時間と精神力を消費する | AIモーニングブリーフィングが毎朝「今日やること」を提示。ダッシュボードで時間帯に応じた情報を適応表示 |
| P-3 | **長期目標との乖離** | 目先のタスクをこなすだけで日々が終わり、長期目標・中期目標が前進しない。いつの間にか「なぜそれをやっているのか」が分からなくなる | 目標階層（長期→中期→週次→日タスク）でタスクと目標を紐づけ。ゴール・ジャーニーマップで可視化。週次レビューでAIが乖離を指摘 |
| P-4 | **スケジュール設計の認知負荷が高すぎる** | 複数PJの締切、習慣、移動時間、エネルギーレベルを考慮したスケジュールを手動で組むのは困難。毎週同じ作業を繰り返している | AIが週間タイムボクシングを自動生成。ユーザーは承認・微調整するだけでよい |
| P-5 | **達成感の欠如** | 一生懸命働いていても「何をやったか」が可視化されず、達成感を得にくい。モチベーションが低下する | Done List自動生成、ストリーク表示、マイルストーン達成時のお祝い演出、ゴール・ジャーニーマップの積み上げ表示 |
| P-6 | **積極的な自己管理の負担** | 習慣トラッキング、振り返り、目標の見直しを自発的に続けるには強い意志が必要。続かなくて当然の設計になっている | AIが積極的に声をかけてくれる（モーニングブリーフィング、デイリークローズ通知、レビュー通知、ナッジ）。ユーザーは「反応するだけ」でよい |
| P-7 | **探索・成長機会の喪失** | 効率化・タスク消化に集中するあまり、新しいことへの挑戦や学習の時間が確保できない | AIがスケジュール生成時に「探索タイム」を自動組み込み。セレンディピティ・エンジンがナレッジ間の意外な繋がりを提示 |
| P-8 | **移動等の現実的制約の未考慮** | デジタルカレンダーは移動時間・準備時間を考慮しないため、非現実的なスケジュールが生成される | Google Calendar連携時に場所情報を取得し、移動時間を自動確保（FR-34） |

## 4. 5つの設計原則

ARDORSの設計はすべて以下の5つの原則に基づく。機能の優先度判断・UXデザイン・AI設計のすべてにおいてこれらを参照する。

### 原則 1: 入力コスト最小化

**「AIが提案し、ユーザーが1タップで承認する」**

ユーザーに長文の入力・細かい設定・手動整理を求めない。AIが草案・提案・整理を行い、ユーザーはそれを承認・修正するだけでよい設計にする。

- タイムボクシングはAIが生成 → ユーザーが承認
- ブレインダンプはAIが構造化 → ユーザーが確認・承認
- モーニングブリーフィングはAIが提示 → ユーザーは読むだけ
- 習慣チェックは1タップで完了

### 原則 2: AI非依存ベースライン

**「AIなしでもすべての機能が動作する」**

AIは体験を強化するレイヤーであり、AIがなければアプリが使えないという状態は作らない。ネットワーク遅延・API障害・コスト超過時でも、手動での操作が常に可能であること。

- タスク・スケジュール・習慣はすべて手動でCRUD可能
- AI提案はオプションであり、スキップできる
- データはローカルDBに保存され、AI応答待ちで処理が止まらない

### 原則 3: 建設的AI（Constructive AI）

**「褒めるだけのAIではなく、データに基づいた率直なフィードバックを行う」**

ユーザーの行動に問題があれば、AIは優しいが率直に指摘する。データが示す事実（目標との乖離・タスク消化率の低下・特定領域への時間集中）を基に、ユーザーに建設的な問いかけを行う。

- 週次レビュー時にAIコーチモードが有効になる
- AIの厳しさレベルはユーザー設定で調整可能（コーチ/メンター/フレンド）
- 指摘は責めず、解決策の提案とセットで行う

### 原則 4: コールドスタート設計

**「初回起動時にAIに必要なコンテキストを与える」**

AIは初日からユーザーのプロジェクト・目標・生活リズムを知った状態でスタートする。そのためオンボーディングで必要な情報を一括収集する。これにより「AIに毎回説明する」という負荷をなくす。

- オンボーディングで収集する情報: PJ一覧、長期目標、生活リズム、優先事項、エネルギーパターン
- 音声入力でスムーズに収集できる設計にする
- 入力後にAIが構造化し確認画面を提示する

### 原則 5: プラットフォーム戦略

**「個人利用からSaaS展開へ、段階的に拡張する」**

フェーズ1は個人利用に最適化した実装を行う。しかし設計段階からマルチユーザー・課金・管理者機能を見据えた構造にする。SuabaseのRLS・認証基盤・管理者画面はSprnt 1から組み込む。

- Supabase RLSにより全データをユーザースコープで隔離
- 管理者画面（SCR-A1, SCR-A2）はSprint 2以降で実装
- 将来的な課金・プラン管理への拡張ポイントを設計に含める

## 7. 成功指標（KPI）

ARDORSは個人利用から始まるが、以下のKPIを測定することで効果を評価し、製品改善に活かす。

### 7.1 生産性・目標達成系

| KPI | 測定方法 | 目標（目安） |
|-----|---------|------------|
| プロジェクト完了数 | ARDORSのPJステータス変化から自動集計 | 使用前比で増加 |
| マイルストーン達成率 | 期限内に完了したマイルストーン数 / 全マイルストーン数 | 70%以上 |
| 週次ゴール達成率 | 週次レビューで「達成」と評価された週次ゴール数 / 全設定数 | 60%以上 |
| 長期目標への時間配分 | タイムボクシングデータから算出した目標別時間配分 | 毎週増加トレンド |

### 7.2 スケジュール・習慣系

| KPI | 測定方法 | 目標（目安） |
|-----|---------|------------|
| スケジュール設計時間の削減 | ユーザー申告（週次レビュー時の問いかけ） | 手動比50%削減 |
| スケジュール遵守率 | タイムブロック完了チェック数 / 全タイムブロック数 | 70%以上 |
| 習慣継続率（ストリーク） | 習慣実行ログから自動算出 | 21日以上連続が1件以上 |

### 7.3 体験・エンゲージメント系

| KPI | 測定方法 | 目標（目安） |
|-----|---------|------------|
| デイリークローズ完了率 | デイリークローズ完了日数 / 利用日数 | 70%以上 |
| ウィークリーレビュー完了率 | ウィークリーレビュー完了週数 / 全利用週数 | 80%以上 |
| 達成感スコア | デイリークローズ時のAI問いかけへの回答（5段階） | 平均3.5以上 |
| アクティブ日数 | 月あたりの利用日数 | 20日以上/月 |

## 8. 対象外スコープ

以下はARDORSのスコープ外とする。設計・実装時にこれらの機能を組み込まないこと。

### 8.1 Phase 1（MVP）での対象外

| カテゴリ | 詳細 | 理由 |
|---------|------|------|
| **チームコラボレーション機能** | 複数ユーザーでのPJ共有、タスクのアサイン、コメント、権限管理 | 個人利用に集中。複雑性が大幅に増加する |
| **ソーシャル機能** | フォロー、いいね、公開プロフィール、他ユーザーとの比較 | 個人の内省を妨げる。プライバシーリスクもある |
| **外部ツール連携（GCal以外）** | Notion・Todoist・Slack・Jira等との双方向同期 | 実装コストが高い。GCalで十分なカバレッジを得られる |
| **バイタルデータ連携** | Apple Health・Fitbit・Garminなどのウェアラブルデバイス連携 | エネルギーチェックインは自己申告で代替。ハードウェア依存を避ける |
| **ネイティブアプリ（iOS/Android）** | スマートフォン専用ネイティブアプリ | WebアプリのレスポンシブでMVPを実現。PMF確認後に検討 |
| **オフラインモード** | ネットワーク未接続時の完全なデータ同期・AI利用 | Webファーストの方針。実装複雑度が高い |
| **課金・サブスクリプション管理** | Stripe連携、プラン管理、課金処理 | Phase 1は個人利用のみ。SaaS展開時に実装 |

### 8.2 永続的スコープ外（将来も含む）

| カテゴリ | 理由 |
|---------|------|
| SNS的なコンテンツ投稿・シェア機能 | コアバリューと相容れない |
| ゲーミフィケーション（ポイント・ランキング） | 外発的動機付けはARDORSの哲学に反する |
| 他ユーザーとのタスク依頼・請負 | フリーランス管理ツールのスコープ |

## 付録: 用語集（主要用語）

| 用語 | 定義 |
|------|------|
| タイムボクシング | 特定の作業に対して固定した時間枠（タイムブロック）を割り当てるスケジュール管理手法 |
| ブレインダンプ | 頭の中にあることをすべて吐き出す作業。ARDORSではAIが内容を自動整理する |
| Active / Warm / Cold | PJの状態分類。Active=今週前進、Warm=止めないが本格着手しない、Cold=明確に保留 |
| コールドスタート | AIが初回起動時にユーザーのコンテキストをまったく知らない状態。オンボーディングで解消する |
| 建設的AI | 褒めるだけでなく、データに基づいて率直に改善点を指摘するAIの設計哲学 |
| P0 / P1 / P2 | 機能優先度。P0=MUST（MVP必須）、P1=SHOULD（体験向上）、P2=COULD（将来差別化） |
| SCR-ID | 画面識別子（例: SCR-20 = ダッシュボード） |
| FR-ID | 機能識別子（例: FR-10 = AIテキスト対話） |
| RLS | Row Level Security。SupabaseのDB行レベルアクセス制御機能 |
| STT | Speech-to-Text（音声認識）。ARDORSではWeb Speech APIを使用 |
