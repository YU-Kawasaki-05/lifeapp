# 受入基準（Acceptance Criteria）
## プロジェクト名: ARDORS（アーダース）
## 対象: P0機能 全件

---

> **記述形式**: Gherkin（Given / When / Then）
> **カバー範囲**: 正常系・境界値・異常系・AI非依存フォールバック
> **用語**: ユーザー = 一般ユーザー（U-01）。管理者は明示

### FR-01: ユーザー登録

```gherkin
Feature: FR-01 ユーザー登録

  # ─── 正常系: Google OAuth ───
  Scenario: Google OAuthで新規登録が成功する
    Given 未登録ユーザーがLP（/）またはログイン画面を表示している
    When 「Googleで登録」ボタンをクリックする
    And GoogleのOAuth認証を許可する
    Then ARDORSのアカウントが作成される
    And オンボーディング画面（SCR-10）に遷移する

  # ─── 正常系: メールアドレス+パスワード ───
  Scenario: メールアドレスとパスワードで新規登録が成功する
    Given 未登録ユーザーが登録画面を表示している
    When 有効なメールアドレスと8文字以上の英字+数字を含むパスワードを入力する
    And 「登録する」ボタンをクリックする
    Then 確認メールが送信される
    And 「確認メールを送信しました。メール内のリンクをクリックして登録を完了してください」と表示される

  Scenario: メール確認リンクをクリックして本登録が完了する
    Given 確認メールを受信しているユーザーがいる
    When メール内の確認リンクをクリックする
    Then アカウントが有効化される
    And オンボーディング画面（SCR-10）に遷移する

  # ─── 異常系: 重複メール ───
  Scenario: 既に登録済みのメールアドレスで登録しようとする
    Given あるメールアドレスで既にアカウントが存在する
    When 同じメールアドレスで新規登録を試みる
    Then 「このメールアドレスは既に使用されています」と表示される
    And アカウントは作成されない

  Scenario: Google OAuthで登録したメールアドレスをメアド+パスワードで再登録しようとする
    Given Google OAuthで登録済みのメールアドレスがある
    When 同じメールアドレスでメアド+パスワード登録を試みる
    Then 「このメールアドレスはGoogleアカウントで登録済みです」と表示される
    And アカウントは作成されない

  # ─── 異常系: パスワード要件未達 ───
  Scenario: パスワードが要件を満たさない
    Given ユーザーが登録フォームを入力している
    When 7文字のパスワード（英字のみ）を入力する
    Then 「パスワードは8文字以上で、英字と数字を含む必要があります」と表示される
    And 登録ボタンは無効のまま

  # ─── 異常系: OAuth失敗 ───
  Scenario: Google OAuth認証がキャンセルまたは失敗する
    Given ユーザーがGoogle OAuthフローに入った
    When Googleの認証画面でキャンセルする
    Then 「Googleアカウントとの連携に失敗しました。再度お試しください」と表示される
    And 登録画面に戻る
```

### FR-02: ログイン / ログアウト

```gherkin
Feature: FR-02 ログイン / ログアウト

  # ─── 正常系: Googleログイン ───
  Scenario: Googleアカウントでログインが成功する
    Given 登録済みのGoogleアカウントユーザーがログイン画面を表示している
    When 「Googleでログイン」ボタンをクリックしOAuth認証を許可する
    Then ダッシュボード（SCR-20）に遷移する
    And セッションが30日間有効になる

  # ─── 正常系: メアド+パスワード ───
  Scenario: メールアドレスとパスワードでログインが成功する
    Given 登録済みのユーザーがログイン画面を表示している
    When 正しいメールアドレスと正しいパスワードを入力して「ログイン」をクリックする
    Then ダッシュボード（SCR-20）に遷移する

  # ─── 正常系: ログアウト ───
  Scenario: ログアウトが成功する
    Given ログイン済みのユーザーが設定画面を表示している
    When 「ログアウト」ボタンをクリックし確認ダイアログで「ログアウトする」を選択する
    Then セッションが破棄される
    And ログイン画面（SCR-03）に遷移する

  # ─── 異常系: 誤認証情報 ───
  Scenario: 誤ったパスワードでログインしようとする
    Given 登録済みのユーザーがログイン画面を表示している
    When 正しいメールアドレスと誤ったパスワードを入力する
    Then 「メールアドレスまたはパスワードが正しくありません」と表示される
    And どちらが間違いかは明示されない（セキュリティ上）

  # ─── 異常系: メール未確認 ───
  Scenario: メール確認が未完了のアカウントでログインしようとする
    Given メアド+パスワードで登録したが確認メールをまだクリックしていないユーザーがいる
    When ログインを試みる
    Then 「メールアドレスの確認が完了していません。確認メールを再送しますか？」と表示される
    And 「再送する」ボタンをクリックすると確認メールが再送される
```

### FR-03: パスワードリセット

```gherkin
Feature: FR-03 パスワードリセット

  # ─── 正常系 ───
  Scenario: パスワードリセットメールを送信する
    Given ユーザーがパスワードリセット画面を表示している
    When 登録済みのメールアドレスを入力して「送信」をクリックする
    Then 「パスワードリセットリンクを送信しました」と表示される
    And 入力したメールアドレスにリセットリンクが届く

  Scenario: リセットリンクから新パスワードを設定する
    Given ユーザーがパスワードリセットメールを受信している
    When メール内のリセットリンクをクリックし（24時間以内）新しいパスワードを設定する
    Then パスワードが更新される
    And 既存の全セッションが無効化される
    And ログイン画面に遷移し「パスワードを変更しました」と表示される

  # ─── 異常系: 未登録メール ───
  Scenario: 未登録のメールアドレスでリセットを要求する
    Given ユーザーがパスワードリセット画面を表示している
    When 未登録のメールアドレスを入力して「送信」をクリックする
    Then 「パスワードリセットリンクを送信しました」と表示される（セキュリティ上、存在の有無を漏洩しない）
    And メールは送信されない

  # ─── 異常系: 期限切れリンク ───
  Scenario: 有効期限切れのリセットリンクをクリックする
    Given パスワードリセットリンクが発行から24時間以上経過している
    When そのリンクをクリックする
    Then 「このリンクは有効期限が切れています。再度リセットを要求してください」と表示される

  # ─── 境界値: 同一パスワード ───
  Scenario: 現在と同じパスワードに変更しようとする
    Given ユーザーがパスワード再設定フォームを表示している
    When 現在と同一のパスワードを入力する
    Then 「現在と同じパスワードは設定できません」と表示される
```

### FR-04: オンボーディング

```gherkin
Feature: FR-04 オンボーディング

  # ─── 正常系: 全ステップ完了 ───
  Scenario: オンボーディングを全ステップ完了する
    Given 新規登録済みのユーザーがオンボーディング画面（SCR-10）を表示している
    When ステップ1でプロジェクト一覧をテキスト入力し「次へ」をクリックする
    And ステップ2で起床・就寝・仕事時間を設定し「次へ」をクリックする
    And ステップ3でゴールと優先事項を入力し「次へ」をクリックする
    And ステップ4でAIが生成した初期データを確認して「開始する」をクリックする
    Then PJ一覧・目標階層の初期データが作成される
    And ダッシュボード（SCR-20）に遷移する

  Scenario: 音声入力でオンボーディングを完了する
    Given ユーザーがオンボーディングのステップ1を表示している
    When マイクボタンをタップし「就活と卒論と個人開発をやってます」と話す
    Then STTがテキストに変換し入力フィールドに表示される
    And AIが3つのPJ候補（就活・卒論・個人開発）に構造化した結果を提示する

  # ─── 正常系: スキップ ───
  Scenario: オンボーディングをスキップする
    Given ユーザーがオンボーディング画面を表示している
    When 「スキップ」をクリックする
    Then ダッシュボード（SCR-20）に遷移する
    And AIのパターン分析機能（FR-85等）は一時的に無効になる
    And 設定画面から後でオンボーディング情報を入力できる

  # ─── 正常系: Google Calendar連携 ───
  Scenario: オンボーディング中にGoogle Calendar連携を設定する
    Given ユーザーがオンボーディングのステップ4を表示している
    When 「Google Calendarと連携する」を選択してOAuth認証を許可する
    Then カレンダーの連携が設定される
    And オンボーディング完了後、GCalの予定がタイムラインに表示される

  # ─── 正常系: Google Calendar連携スキップ ───
  Scenario: Google Calendar連携をスキップする
    Given ユーザーがオンボーディングのステップ4を表示している
    When 「後で設定する」を選択する
    Then GCal連携なしでオンボーディングが完了する
    And 設定画面からいつでも連携できる
```

### FR-10: AIテキスト対話

```gherkin
Feature: FR-10 AIテキスト対話

  # ─── 正常系: タスク作成依頼 ───
  Scenario: AIへの依頼でタスクが作成される
    Given ログイン済みのユーザーがAI対話画面（SCR-21）を表示している
    When 「就活のES提出を来週金曜までのタスクに追加して」とテキスト送信する
    Then AIが「就活PJに『ES提出』（期限: 来週金曜）を追加します。よいですか？」と承認確認を提示する
    And 「OK」をクリックするとタスクが作成される
    And 「キャンセル」をクリックするとタスクは作成されない

  # ─── 正常系: 次の行動提案 ───
  Scenario: AIが次の行動を提案する
    Given ユーザーがAI対話画面を表示している
    When 「今日何をすべきか教えて」と送信する
    Then AIが今日のスケジュール・未完了タスク・期限を考慮した行動提案を返す
    And 提案には理由が付く（建設的AIの原則）

  # ─── 正常系: 承認なし変更は発生しない ───
  Scenario: AIの提案をキャンセルするとデータは変更されない
    Given AIがタスク作成の承認確認を提示している
    When ユーザーが「キャンセル」をクリックする
    Then タスクは作成されない
    And データへの変更は一切発生しない

  # ─── 正常系: 対話履歴の継続 ───
  Scenario: 前の対話内容を踏まえた回答が返る
    Given ユーザーが「就活のESを書くのが大変」と送信した対話履歴がある
    When 「どこから手をつければいい？」と追加送信する
    Then AIが前の発言（ES執筆への苦手感）をコンテキストとして踏まえた回答を返す

  # ─── 異常系: LLM API障害 ───
  Scenario: LLM APIが利用不能な場合にフォールバックする
    Given LLM APIが応答不能である
    When ユーザーがAI対話にテキストを送信する
    Then 「現在AIが利用できません。手動で操作してください」と表示される
    And 手動タスク作成・PJ管理への導線が表示される

  # ─── 異常系: 入力長超過 ───
  Scenario: 10,000文字を超えるテキストを送信しようとする
    Given ユーザーがAI対話画面を表示している
    When 10,001文字のテキストを入力して送信しようとする
    Then 「入力が長すぎます。分割して送信してください」と表示される
    And 送信は実行されない
```

### FR-11: 音声入力（STT）

```gherkin
Feature: FR-11 音声入力（STT）

  # ─── 正常系 ───
  Scenario: 音声をテキストに変換してAIに送信する
    Given ユーザーがAI対話画面を表示しマイク権限を許可済みである
    When マイクボタンをタップし「明日の午前中に卒論の序論を書く」と話す
    And マイクボタンを再度タップして録音を停止する
    Then 「明日の午前中に卒論の序論を書く」というテキストが入力フィールドに表示される
    And ユーザーは送信前にテキストを確認・修正できる
    And 「送信」をタップするとAI対話（FR-10）として処理される

  # ─── 境界値: 5分録音 ───
  Scenario: 音声録音が5分に達すると自動停止する
    Given ユーザーが音声録音中である
    When 録音開始から5分が経過する
    Then 録音が自動停止する
    And 「最大録音時間（5分）に達しました」と表示される
    And それまでの音声がSTT変換される

  # ─── 異常系: マイク権限なし ───
  Scenario: マイク権限が未許可の状態でマイクボタンをタップする
    Given ブラウザのマイク権限が未許可または拒否されている
    When マイクボタンをタップする
    Then ブラウザの権限リクエストダイアログが表示される
    And 拒否された場合「マイクへのアクセスが許可されていません。ブラウザの設定を確認してください」と表示される

  # ─── 異常系: STT変換失敗 ───
  Scenario: 音声が認識できなかった場合
    Given ユーザーが録音を停止した
    When STTが音声を認識できなかった
    Then 「音声を認識できませんでした。再度お試しください」と表示される

  # ─── 異常系: STT API障害 ───
  Scenario: STT APIが利用不能な場合
    Given STT APIが応答不能である
    When ユーザーがマイクボタンをタップする
    Then 「音声入力が利用できません。テキストで入力してください」と表示される
    And テキスト入力フィールドにフォーカスが移動する
```

### FR-12: ブレインダンプ

```gherkin
Feature: FR-12 ブレインダンプ

  # ─── 正常系: テキスト入力 → AI構造化 ───
  Scenario: 混在した内容を送信するとAIが構造化して提示する
    Given ユーザーがAI対話画面を表示している
    When テキストで「就活の面接準備しなきゃ、来週論文の締切、新しいアプリのアイデアが浮かんだ」と入力する
    Then AIがタスク候補（面接準備、論文対応）と新規PJ候補（新アプリ）を構造化して提示する
    And 各候補に承認/却下/修正ボタンが表示される
    And データはまだ保存されていない

  # ─── 正常系: 承認 → データ反映 ───
  Scenario: 構造化結果を承認するとデータに反映される
    Given AIが構造化結果を提示している
    When ユーザーがタスク候補「面接準備」を承認する
    Then 就活PJにタスク「面接準備」が作成される
    And 承認済み項目のステータスが「反映済み」に変わる

  # ─── 正常系: 部分承認 ───
  Scenario: 一部のみ承認する
    Given AIが3件の候補（タスク2件・新規PJ1件）を提示している
    When ユーザーがタスク1件を承認し、タスク1件を却下し、PJ候補を修正して承認する
    Then 承認・修正承認した2件のみがデータに反映される
    And 却下した1件はデータに反映されない

  # ─── 正常系: 新規PJ候補の検出 ───
  Scenario: ブレインダンプで新規PJが提案される
    Given ユーザーが「教育系のサービスを作りたいと思ってる」とブレインダンプする
    When AIが新規PJ候補を検出する
    Then AIが「新しいプロジェクト：教育サービス開発」として、ゴール・マイルストーン・初期タスクを提案する
    And ユーザーが承認するとFR-24（新規PJ作成）のフローに接続する

  # ─── 異常系: LLM API障害 ───
  Scenario: LLM APIが利用不能な場合
    Given LLM APIが応答不能である
    When ユーザーがブレインダンプのテキストを送信する
    Then 「現在AIが利用できません。手動で操作してください」と表示される
    And 手動タスク作成画面への導線が表示される
```

### FR-13: AIモーニングブリーフィング

```gherkin
Feature: FR-13 AIモーニングブリーフィング

  # ─── 正常系: 朝のダッシュボード表示 ───
  Scenario: 朝にダッシュボードを開くとブリーフィングが表示される
    Given ログイン済みのユーザーがその日初めてダッシュボードにアクセスする
    And 現在時刻が生活リズム設定の「起床時間±2時間」の範囲内である
    Then AIが今日のスケジュール・最重要タスク・昨日からの接続・締切アラートを含むブリーフィングを表示する
    And ブリーフィングは行動につながる具体的な内容である（単なる励ましではない）

  Scenario: 昨日のデイリークローズ内容を踏まえたブリーフィングが表示される
    Given 昨日のデイリークローズで「卒論の序論でつまずいている」と記録されている
    When 翌朝ダッシュボードを開く
    Then 「昨日『卒論の序論でつまずいていた』とのこと。今日は序論の第1段落だけに集中してみましょう」のような接続コメントが含まれる

  # ─── 正常系: 1日1回のみ ───
  Scenario: 同日中に2回目以降のダッシュボードアクセスではブリーフィングは再表示されない
    Given ユーザーが今日すでにダッシュボードを開いてブリーフィングを受け取った
    When 再度ダッシュボードにアクセスする
    Then ブリーフィングは表示されない
    And 通常のダッシュボード表示のみ

  # ─── 正常系: コールドスタート期間 ───
  Scenario: データが少ないコールドスタート期間のブリーフィング
    Given ユーザーがARDORSを使い始めて3日目以内（データ蓄積が少ない）
    When 朝にダッシュボードを開く
    Then 基本的なスケジュール概要のみが表示される
    And パターン分析に基づく提案は含まれない

  # ─── 異常系: LLM API障害 ───
  Scenario: LLM APIが利用不能でもダッシュボードは表示される
    Given LLM APIが応答不能である
    When ユーザーが朝にダッシュボードを開く
    Then AIブリーフィングは表示されない
```

## 付録: 共通ルール

```gherkin
Feature: 共通: AI非依存フォールバック

  Scenario Outline: LLM API障害時に手動操作への導線が表示される
    Given LLM APIが応答不能である
    When ユーザーが<AI依存機能>を使用しようとする
    Then 「現在AIが利用できません。手動で操作してください」と表示される
    And 対応する手動操作（CRUD画面等）への導線が表示される

    Examples:
      | AI依存機能               |
      | AIテキスト対話           |
      | ブレインダンプ           |
      | AIモーニングブリーフィング|
      | 週次タイムボクシング生成  |
      | ウィークリーレビューAI分析|
      | デイリークローズAIフィードバック |

Feature: 共通: 建設的AIの原則

  Scenario: AIは満足させる言葉だけでなく客観的視点を提供する
    Given ユーザーがAIに現在の状況を共有した
    When AIが応答する
    Then 応答は「目的達成に向けた客観的な視点」を含む
    And データや実績に基づいた具体的な内容である
    And 単なる励ましや同意だけで終わることはない

Feature: 共通: ユーザー承認の原則

  Scenario: AIはユーザーの明示的な承認なしにデータを変更しない
    Given AIがタスク/PJ/スケジュールの変更を提案した
    When ユーザーがまだ承認していない
    Then データへの変更は実行されない
    And 承認/却下ボタンが常に表示されている
```
